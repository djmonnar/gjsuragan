# 리뉴얼 전 안전성 기준

이 문서는 실제 영업에 사용 중인 궁중수라간 주문·배송·월식 관리 시스템의 리뉴얼 전 안전성 기준을 기록한다. 운영 기준 브랜치는 `main`이며, 이번 안전성 작업의 기준 커밋은 `33ae90f58becec603aad3bd3d0831e1674c3361a`이다.

이번 작업은 화면 리뉴얼이나 데이터 마이그레이션이 아니다. 기존 기능과 데이터 모델을 유지하면서 인증, 배송 처리, 외부 문자열 출력, 가입 권한 및 최소 테스트 기반만 보강한다.

## 현재 운영 구조

- 직원용 배송관리 페이지는 Firebase 이메일/비밀번호 인증 후 허용된 관리자 계정만 업무 화면과 Firestore 연결을 시작한다.
- 관리자 페이지도 Firebase Authentication을 기반으로 운영된다.
- 현재 직원은 사무 주문 관리와 실제 배송 업무를 함께 수행하므로, 직원 페이지의 배송완료·취소 흐름은 동일한 `customers/{customerId}` 문서를 갱신한다.
- 별도 배송지도 페이지는 Firebase 익명 인증 세션으로 직배송 목록을 읽고 완료 상태를 기록한다. 이 구조는 이번 작업에서 변경하지 않았다.
- 행사도시락 주문 페이지는 공개 주문 입력 화면이며 `eventOrders`에 요청을 저장한다. 공개 주문 구조와 권한 모델은 이번 작업에서 변경하지 않았다.
- 프론트는 Firebase compat SDK를 사용한다. 직원/배송지도는 10.12 계열, 관리자·고객·행사 주문은 9.22 계열 스크립트를 사용하며 이번 작업에서 버전을 변경하지 않았다.
- Functions 런타임은 Node.js 22이다.
- 저장소에는 자동 배포 GitHub Actions 워크플로가 없다. Firebase 설정은 `firebase.json`, `firestore.rules`, `firestore.indexes.json`, `storage.rules`, `functions/`에 있다.

## 이번 안전성 보강

### 카카오 인증

`handleKakaoWebhookRequest()` 진입 직후, OCR·업무 데이터 조회·배송 명령보다 앞에 선택형 인증 게이트를 연결했다.

- 기능 플래그: `KAKAO_AUTH_ENFORCEMENT`
- 기본값: 비활성
- PIN 설정: `KAKAO_ADMIN_PIN`
- 선택 화이트리스트: `KAKAO_ALLOWED_USERS`
- 인증 세션: 기존 `kakaoBotSessions` 컬렉션과 기존 만료시간을 유지

플래그가 없거나 거짓이면 기존 카카오 동작을 유지한다. 플래그가 참인데 PIN 설정이 없거나 인증되지 않은 요청이면 안전하게 거부한다. 화이트리스트가 설정돼 있으면 목록에 포함된 사용자만 PIN 인증을 진행할 수 있다.

### 배송완료와 취소

직원 페이지, 배송관리 일괄 처리, 배송지도에서 공통 트랜잭션 helper를 사용한다.

- 트랜잭션 안에서 `customers/{customerId}` 최신 문서를 다시 읽는다.
- 같은 날짜 완료는 한 번만 차감한다.
- 잔여 횟수는 음수가 되지 않는다.
- 완료 기록이 있는 날짜만 취소하고 한 번만 복구한다.
- 기존 `deliveredDates`, `remain`, `status` 필드와 날짜 형식을 유지한다.
- 직원 화면에서 선택주문을 한 번에 종료하는 기존 규칙은 해당 호출에서만 유지한다.
- 기존 완료 메타데이터(`lastDeliveredDate`, `deliveredAt`, `updatedAt`, `deliveryState`)도 기존 직원 흐름에서 유지한다.

### 외부 문자열 출력

기존 `escHtml()`을 재사용해 다음 렌더링 경로의 외부 문자열을 문자 그대로 표시한다.

- 직원 대시보드와 배송관리 고객 정보
- 고객 상세 및 배송 인쇄 목록
- 아임웹 주문 미리보기
- 텍스트 붙여넣기 미리보기
- 엑셀 가져오기 미리보기

정적 버튼, 아이콘, 배지, 화면 구조는 변경하지 않았다. 전체 렌더링 엔진을 재작성하지 않았으므로 다른 경로의 전수 정리는 별도 작업으로 다룬다.

### 가입 가격 필드

고객 가입 payload에서 `lunchPrice`, `saladPrice`, `priceLunch`, `priceSalad`를 제거하고 일반 고객의 `users` 문서 생성 허용 목록에서도 제외했다.

- 고객은 가입 시 또는 이후 수정으로 가격 필드를 쓸 수 없다.
- 기존 관리자 쓰기 권한은 유지한다.
- 기존 고객 문서는 변경하거나 마이그레이션하지 않는다.
- 가격 필드가 아직 없는 신규 고객 화면은 기존 기본 단가를 사용한다.

## 테스트 실행

Functions 디렉터리에서 실행한다.

```bash
cd functions
npm install
npm run lint
npm run test:unit
npm run test:emulator
npm run test:smoke
```

전체 검사는 다음과 같다.

```bash
npm test
```

Firestore Emulator에는 Java 21 이상이 필요하다. 테스트는 `demo-gjsuragan-safety` 프로젝트 ID와 로컬 Emulator만 사용하며 운영 Firebase에 연결하지 않는다.

테스트 범위:

- 카카오 플래그 기본 비활성, 잘못된 PIN, 정상 PIN, 세션 만료, 화이트리스트, 설정 누락
- 인증 게이트가 OCR·조회·배송 명령보다 먼저 실행되는지 확인
- 배송 완료/취소 단위 테스트와 동시 트랜잭션 테스트
- 가입 가격 필드 Firestore Rules Emulator 테스트
- 지정된 HTML/XSS 문자열 escape 테스트
- 직원, 관리자, 고객, 배송지도, 행사 주문 페이지와 로컬 스크립트 문법 스모크 검사

## 운영 활성화 전 설정

실제 값은 저장소나 문서에 기록하지 않는다.

1. `KAKAO_ADMIN_PIN`을 Functions 비밀 설정 또는 현재 프로젝트의 안전한 환경설정 방식으로 등록한다.
2. 필요하면 `KAKAO_ALLOWED_USERS`에 허용할 카카오 사용자 ID를 콤마로 구분해 등록한다.
3. `KAKAO_AUTH_ENFORCEMENT`는 먼저 비활성 상태로 배포해 기존 명령을 확인한다.
4. 별도 검증 환경에서 플래그를 활성화하고 미인증 차단, PIN 인증, 세션 만료, OCR, 고객조회, 배송 명령을 확인한다.
5. 운영 활성화는 담당자가 수동 승인한 뒤 진행한다.

## 권장 운영 배포 순서

이번 브랜치에서는 실제 배포하지 않는다. 사람이 승인한 뒤 다음 순서를 권장한다.

1. 현재 운영 `main` 커밋 SHA와 Firestore 백업 상태를 기록한다.
2. 별도 staging 또는 Emulator에서 전체 테스트를 다시 실행한다.
3. 카카오 인증 플래그 비활성 상태로 Functions 변경을 검증한다.
4. 정적 페이지 변경을 검증하고 직원/배송지도 완료·취소를 수동 확인한다.
5. 가입과 관리자 가격 설정을 검증한 뒤 Firestore Rules 적용 여부를 승인한다.
6. 카카오 PIN·화이트리스트를 준비한 후 별도 검증 환경에서 인증 플래그를 활성화한다.
7. 직원 실사용 검증이 끝난 뒤 운영 활성화 여부를 최종 승인한다.

컴포넌트는 Functions, 정적 페이지, Firestore Rules를 각각 검증하고 한 번에 묶어 배포하지 않는 것을 권장한다.

## 롤백 기준

- Git: 기준 운영 SHA 또는 배포 직전 태그로 정적 파일과 Functions 소스를 되돌린다. 강제 push 대신 명시적인 revert 커밋을 사용한다.
- Functions: 이전 정상 리비전과 환경설정 이름을 확인한 후 이전 코드 리비전으로 복원한다.
- Firestore Rules: 배포 직전 저장한 Rules 버전으로 되돌리고 고객 가입과 관리자 가격 쓰기를 다시 확인한다.
- 정적 페이지: 이전 정상 커밋의 HTML/JS만 복원하고 캐시 영향 여부를 확인한다.
- 카카오: 인증 관련 장애가 있으면 `KAKAO_AUTH_ENFORCEMENT`를 거짓으로 돌려 기존 동작으로 우선 복귀한다.
- 배송: 완료/취소 문제가 있으면 `delivery-transaction.js`를 참조하기 전 코드로 revert하고, 해당 기간의 중복 완료 여부를 운영자가 확인한다.

실제 롤백과 운영 데이터 수정은 이 문서 작업에서 수행하지 않는다.

## 보류 및 추가 확인 사항

- 이번 작업은 우선순위 렌더링 경로만 escape했다. 전체 `innerHTML` 전수 교체는 별도 PR이 필요하다.
- 브라우저 자동화 도구를 새로 추가하지 않았으므로 페이지 검사는 HTML/JavaScript 문법 스모크 수준이다. 실제 기기와 로그인 세션을 포함한 수동 검증이 필요하다.
- 배송지도 익명 인증과 행사도시락 공개 주문의 보안 모델은 운영 영향이 커 이번 범위에서 변경하지 않았다.
- Functions 의존성 감사에서 중간/높음 등급의 기존 전이 의존성 경고가 남아 있다. 자동 `npm audit fix`는 런타임 변경 위험 때문에 실행하지 않았으며 별도 검토가 필요하다.
- 카카오 인증 플래그는 기본 비활성이다. 운영 설정과 수동 확인 없이 활성화하지 않는다.
